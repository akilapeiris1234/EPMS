### Table 2: `IncomingPackages`

Every package that **arrives** at the building. The **Incoming tag** (green badge in UI) comes from the fact that this record is in this table.

> 💡 **Two UI tags explained:**
> - 🟢 **Incoming / Outgoing tag** = which TABLE the record is in (`IncomingPackages` = Incoming, `OutgoingPackages` = Outgoing)
> - 🔵 **Single / Batch tag** = the `Mode` column (`'single'` or `'batch'`)

| Column | Type | Notes |
|--------|------|-------|
| `Id` | INT PK IDENTITY | Auto-increment |
| `TrackingNumber` | NVARCHAR(100) | Unique, scanned barcode |
| `ReferenceNumber` | NVARCHAR(100) | Auto-generated (`YYYYMMDD-HHMM-00001`) |
| `Mode` | NVARCHAR(20) | 🔵 **Single/Batch tag** — `'single'` or `'batch'` |
| `CustomerName` | NVARCHAR(150) | Who receives the package |
| `Time` | NVARCHAR(50) | Time of arrival |
| `Date` | NVARCHAR(50) | Date of arrival |
| `EmployeeId` | INT | Employee receiving it |
| `EmployeeName` | NVARCHAR(150) | |
| `EmployeeCompany` | NVARCHAR(150) | |
| `Department` | NVARCHAR(100) | |
| `DeliveryCompany` | NVARCHAR(150) | Courier company name |
| `DeliveryPersonName` | NVARCHAR(150) | Who delivered it |
| `VehicleNumber` | NVARCHAR(50) | Plate number |
| `VehicleType` | NVARCHAR(100) | Van / Car / Motorcycle / Pickup |
| `Remark` | NVARCHAR(1000) | Additional notes about the package |
| `VerificationStatus` | NVARCHAR(50) | Default `'Pending'` |
| `HoldingState` | BIT | 0=normal, 1=on hold |
| `HoldingReason` | NVARCHAR(500) | Why it is on hold |
| `GuardVerificationStatus` | NVARCHAR(50) | `'verified'` or `'pending'` |
| `GuardId` | NVARCHAR(100) | Guard's access ID |
| `GuardName` | NVARCHAR(150) | Guard's name |
| `GuardVerifiedAt` | DATETIME | Nullable |
| `VerifiedAt` | DATETIME | Nullable |
| `CreatedAt` | DATETIME | Default `GETDATE()` |
| `UpdatedAt` | DATETIME | Default `GETDATE()` |

**Fields used in Single Incoming Package form:**
- `trackingNumber` ✅ required
- `customerName` ✅ required
- `mode` = `'single'`
- `employeeName` (optional)
- `deliveryCompany`, `deliveryPersonName`, `vehicleNumber`, `vehicleType`
- `remark` (optional notes)

**Extra fields used in Batch Incoming Package form:**
- `referenceNumber` (auto-generated, shared across all packages in batch)
- `mode` = `'batch'`
- `batchId` = same as `referenceNumber`
- Multiple `trackingNumber` entries — one row per package

**Verification form adds:**
- `department`, `employeeId`, `guardId`, `guardStatus` (`'verified'` or `'pending'`)

### Table 4: `GateRecords`

Stores every entry/exit gate event. Replaces `localStorage` used in the current `EntryExitRecording` page.

| Column | Type | Notes |
|--------|------|-------|
| `Id` | INT PK IDENTITY | Auto-increment |
| `PersonnelId` | NVARCHAR(100) | Scanned ID card value (employee ID or visitor ID) |
| `Name` | NVARCHAR(150) | Employee name or visitor name |
| `Category` | NVARCHAR(100) | Department (if employee) or reason for visit (if visitor) |
| `Type` | NVARCHAR(50) | `'employee'` or `'visitor'` |
| `ExitTime` | NVARCHAR(50) | Time the person left |
| `ReturnTime` | NVARCHAR(50) | Time the person returned (NULL = still outside) |
| `Date` | NVARCHAR(50) | Date of the record |
| `CreatedAt` | DATETIME | Default `GETDATE()` |

---

### Table 5: `AccessIds`

Stores every barcode access ID that is generated and printed for users (guards, admins, employees). Replaces the current in-memory/localStorage generation.

| Column | Type | Notes |
|--------|------|-------|
| `Id` | INT PK IDENTITY | Auto-increment |
| `AccessId` | NVARCHAR(50) | Generated ID, e.g. `GRD2604001234` — **unique** |
| `Name` | NVARCHAR(150) | Full name of person |
| `Username` | NVARCHAR(100) | Login username — **unique** |
| `Type` | NVARCHAR(50) | `'user'`, `'admin'`, `'guard'`, `'supervisor'` |
| `IsActive` | BIT | 1=active, 0=revoked |
| `IssuedAt` | DATETIME | When the ID was created — Default `GETDATE()` |
| `RevokedAt` | DATETIME | NULL = still active; set when ID is revoked |

**Access ID format** (from `accessIdGenerator.ts`):
- Guard: `GRD` + `YYMM` + `0001` + `123` + checksum digit → e.g. `GRD260400011234`
- Admin: `ADM...`
- Supervisor: `SUP...`
- Employee/User: `USR...`

---

### Table 6: `Employees`

Stores registered employees. Used by `EmployeeBarcodePrintCard` to print employee ID cards with barcodes.

| Column | Type | Notes |
|--------|------|-------|
| `Id` | INT PK IDENTITY | Auto-increment |
| `EmployeeId` | NVARCHAR(50) | Generated ID e.g. `EMP2604000125` — **unique**, barcode value |
| `EmployeeName` | NVARCHAR(150) | Full name |
| `EmployeeCompany` | NVARCHAR(150) | Which company they belong to |
| `Department` | NVARCHAR(100) | Which department |
| `IsActive` | BIT | 1=active, 0=inactive |
| `CreatedAt` | DATETIME | Default `GETDATE()` |

**Employee ID format** (from `employeeIdGenerator.ts`):
- `EMP` + `YYMM` + `00001` + `25` + checksum digit → e.g. `EMP260400001251`

## 🔗 How the System Works (Data Flow)

```
User fills form (page.tsx)
        ↓
Validation check
        ↓
GuardSelectionModal: "Is guard available?"
    ├── YES → GuardIdModal: guard enters their ID
    │          ↓ submits with guardId, time, date, guardStatus="verified"
    └── NO  → HoldingStateModal: package placed on hold
               ↓ submits with guardStatus="pending", HoldingState=1
        ↓
API call → app/api/route.ts (currently empty - needs to be built)
        ↓
lib/db.js → SQL Server: GuardSystemDB
        ↓
INSERT into IncomingPackages or OutgoingPackages table
```

## 📝 Important Notes

1. **`app/api/route.ts` is empty** — all form submits currently just `console.log()` the data. No data is saved to the DB yet. You need to build API routes to INSERT into the DB.

2. **`GateRecords` table is now in the schema** — but the `EntryExitRecording` page still uses `localStorage`. The page needs to be updated to call an API route that INSERTs into `GateRecords` instead.

3. **`AccessIds` table is now in the schema** — but the `AddUser` page currently generates the ID in memory only. It needs to be updated to save the AccessId to the DB via an API route after the user is created.

4. **`Employees` table is now in the schema** — but the `AddEmployee` admin page needs to be updated to save to DB and generate a barcode via `EmployeeBarcodePrintCard`.



6. **Dashboard and package list pages** show hardcoded sample data, not real DB data. Connect them to the DB via API routes.

7. **`Mode` column** (`'single'` or `'batch'`) — this is the **Incoming/Outgoing tag** shown in the UI on every package row. It is now stored in both `IncomingPackages` and `OutgoingPackages` tables.


9. **`VerificationStatus`** values in use: `'Pending'`, `'Collected'`, `'Cancelled'`, `'Completed'`

10. **`HoldingState = 1`** means the package is flagged and needs guard verification later. The `VerifyHoldingPackages` page handles this.

11. **Folder name typo**: The dashboard folder is named `Dashbaord` (not `Dashboard`) — this is intentional in the current code.
